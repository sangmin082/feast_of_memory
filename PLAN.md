# 기억의 만찬 (Feast of Memory) — iOS 게임 개발 플랜

> 데스게임 스릴러 × 기억 퍼즐 × 인터랙티브 내러티브
> 플랫폼: iOS (iPhone 우선, iPad 대응)

---

## 1. 게임 컨셉

### 1.1 하이 컨셉 (One-liner)

**"기억을 잃은 8인이 초대된 만찬. 매 코스가 끝날 때마다 한 명이 사라진다. 살아남으려면 '기억'을 걸고 게임에서 이겨야 한다."**

### 1.2 핵심 아이디어

- 플레이어는 기억을 잃은 채 낯선 저택의 만찬장에서 깨어난다.
- 만찬은 총 **7개의 코스(라운드)**로 진행되며, 각 코스마다 '기억'을 판돈으로 거는 데스게임이 벌어진다.
- 게임에서 지면 기억을 빼앗기고, 기억을 모두 잃은 자는 만찬장에서 "퇴장"당한다.
- 게임 속 미니게임(기억 퍼즐)과 캐릭터들과의 대화·추리를 통해 **자신이 누구인지, 왜 이곳에 왔는지, 주최자가 누구인지**를 밝혀내는 것이 최종 목표.
- 회수한 기억 조각이 곧 스토리 단서이자 게임 자원이라는 점이 핵심 훅(hook).

### 1.3 장르 & 레퍼런스

| 요소 | 레퍼런스 |
|---|---|
| 데스게임 분위기·긴장감 | 아이의 거짓말(Liar Game), 오징어 게임, 단간론파 |
| 추리 + 어드벤처 파트 | 역전재판, 라이프 이즈 스트레인지 |
| 기억 퍼즐 미니게임 | 카드 매칭류 + 리듬/순발력 변형, The Room 류의 오브젝트 퍼즐 |
| 멀티 엔딩 내러티브 | 지하철의 그녀, 서머타임 렌더 스타일 루프물 |

### 1.4 타깃

- 코어 타깃: 스토리 게임·추리물·데스게임 콘텐츠를 좋아하는 15~35세
- 세션 길이: 챕터당 20~30분 (모바일 친화적 분절 구조)
- 등급: 15세 이상 (직접적 고어 없이 연출로 긴장감 표현 → 앱스토어 심의 리스크 최소화)

## 2. 게임 구조 & 핵심 메카닉

### 2.1 게임 루프

```
[코스 시작]
   ↓
① 만찬 파트 — 캐릭터들과 대화, 단서 수집, 신뢰도 관리 (어드벤처/비주얼노벨)
   ↓
② 게임 선언 — 주최자 '집사'가 이번 코스의 데스게임 규칙 공개
   ↓
③ 데스게임 파트 — 기억 칩을 걸고 미니게임 + 심리전 (배팅/블러핑)
   ↓
④ 정산 파트 — 승자는 기억 조각 획득(스토리 회상 씬 해금), 패자는 기억 상실
   ↓
⑤ 퇴장 이벤트 — 기억이 바닥난 캐릭터의 퇴장 연출, 관계도 변화
   ↓
[다음 코스] … 7코스 후 진엔딩 분기
```

### 2.2 핵심 자원: 기억 칩 (Memory Chip)

- 모든 캐릭터는 기억 칩 N개를 가지고 시작. 칩 = 목숨이자 화폐.
- 칩을 잃으면 실제로 **플레이어의 UI에서 정보가 지워진다**
  (예: 캐릭터 이름이 "???"로 바뀜, 이미 본 대화 로그가 흐려짐, 지도 일부가 안개 처리).
  → "기억을 잃는다"를 시스템으로 체감시키는 시그니처 연출.
- 칩을 획득하면 회상 씬(스토리 조각)이 해금되어 진상에 접근.

### 2.3 데스게임(미니게임) 라인업 — 7코스

각 코스는 "기억"을 테마로 한 변형 게임. 모두 **운 + 실력 + 심리전**이 섞이도록 설계.

| 코스 | 게임 | 메카닉 | 심리전 요소 |
|---|---|---|---|
| 1 (Apéritif) | 기억의 잔 | 셔플되는 잔 추적 (셸 게임 변형) | 상대의 시선 유도 대사 선택 |
| 2 (Soupe) | 거짓의 수프 | 진술 카드 중 거짓 찾기 (라이어 게임식) | NPC 신뢰도에 따라 힌트 왜곡 |
| 3 (Poisson) | 침묵의 경매 | 기억 칩 비공개 배팅 경매 | 블러핑, 칩 잔량 추리 |
| 4 (Entrée) | 만찬석 배열 | 순서 기억 퍼즐 (사이먼 게임 고난도 변형) | 방해 연출(조명, 소음) |
| 5 (Sorbet) | 반쪽의 초상 | 조각난 초상화 복원 (직소+기억력) | 제한시간 거래 가능 |
| 6 (Rôti) | 러시안 서비스 | 독이 든 접시 피하기 (확률+단서 추리) | 접시 교환 협상 |
| 7 (Dessert) | 최후의 만찬 | 1:1 최종전 — 앞선 6코스 메카닉 총집합 | 최종 보스와의 두뇌 대결 |

### 2.4 어드벤처 파트 시스템

- **대화 시스템**: 선택지 기반. 선택에 따라 캐릭터별 신뢰도(±) 변동.
- **신뢰도**: 데스게임 중 동맹/배신 이벤트, 힌트 제공 여부, 생존 분기에 영향.
- **단서 수첩**: 수집한 단서·기억 조각을 정리하는 인게임 노트. 특정 조합 시 "추리 확정" 이벤트.
- **탐색**: 코스 사이 쉬는 시간에 저택 일부 구역 탐색 가능 (포인트 앤 클릭).

### 2.5 엔딩 분기

- **배드 엔딩**: 기억 칩 소진 → 퇴장 (챕터 리트라이 가능)
- **노멀 엔딩**: 7코스 생존, 그러나 진상 미해명
- **트루 엔딩**: 핵심 기억 조각 12개 이상 + 특정 캐릭터 신뢰도 조건 충족 → 만찬의 진실 공개
- **히든 엔딩**: 2회차 전용 — 1회차에서 알 수 없는 단서를 활용 (회차 플레이 유도)

## 3. 스토리 & 캐릭터 (개요)

### 3.1 등장인물 8인 (초안)

| 캐릭터 | 아키타입 | 비밀 |
|---|---|---|
| 주인공 (플레이어) | 기억상실자 | 사실 이 만찬과 가장 깊게 얽힌 인물 |
| 집사 | 진행자/호스트 | 주최자의 대리인? 혹은 주최자 본인? |
| 의사 | 이성적 조력자 | 참가자 중 유일하게 기억이 온전함 |
| 배우 | 매력적 거짓말쟁이 | 블러핑 특화, 초반 최대 경계 대상 |
| 노신사 | 과거의 증인 | 저택의 옛 주인과 관련 |
| 소녀 | 순수/미스터리 | 만찬에 어울리지 않는 존재, 열쇠 캐릭터 |
| 형사 | 추리 라이벌 | 플레이어를 의심하며 추적 |
| 요리사 | 침묵의 관찰자 | 매 코스의 요리에 단서를 숨김 |

### 3.2 스토리 제작 산출물

- 세계관 바이블 1부 (배경, 만찬의 규칙, 주최자의 목적)
- 챕터별 시나리오 7편 + 프롤로그/에필로그
- 캐릭터별 회상 씬 스크립트 (기억 조각 24개 분량)
- 분기 플로우차트 (엔딩 4종)

## 4. 기술 스택 & 아키텍처

### 4.1 스택 선정

| 영역 | 선택 | 이유 |
|---|---|---|
| 언어 | **Swift 5.10+** | iOS 네이티브 최적 |
| UI 프레임워크 | **SwiftUI** (메뉴/대화/수첩 UI) + **SpriteKit** (미니게임/연출) | 비주얼노벨 파트는 SwiftUI가 생산성 높고, 미니게임은 SpriteKit이 애니메이션·파티클에 유리. `SpriteView`로 자연스럽게 혼합 가능 |
| 미니게임 대안 | 필요 시 특정 게임만 Metal 셰이더 활용 | "기억이 지워지는" 글리치/디졸브 연출 |
| 데이터 | 스토리·대화: **JSON/Yarn 형식 스크립트** + 코드 생성 / 세이브: **SwiftData** | 시나리오 라이터가 코드 없이 작업 가능하게 분리 |
| 오디오 | AVFoundation (+ 코스별 어댑티브 BGM 레이어링) | |
| 최소 지원 | iOS 17+ | SwiftData, SwiftUI 최신 API 활용 |
| 아키텍처 | MVVM + 유한상태기계(FSM) 게임 플로우 | 코스 진행(§2.1 루프)이 상태 기계와 정확히 일치 |
| 의존성 | 최소화 (필요 시 SPM만) | |

### 4.2 모듈 구조 (SPM 멀티 모듈)

```
FeastOfMemory (App)
├── CoreKit          — 게임 상태 FSM, 기억 칩 경제, 세이브/로드
├── StoryKit         — 시나리오 파서, 대화 엔진, 분기/플래그 관리
├── DialogueUI       — 비주얼노벨 UI (대화창, 선택지, 캐릭터 스탠딩)
├── MiniGames        — 코스별 미니게임 (게임당 서브 타깃 1개)
│   ├── ShellGame, LiarSoup, SilentAuction, ...
├── MemoryFX         — "기억 소실" 연출 (블러, 글리치, 이름 마스킹)
├── AudioKit         — BGM/SFX 매니저
└── PersistenceKit   — SwiftData 세이브, iCloud 동기화(CloudKit)
```

### 4.3 핵심 시스템 설계 포인트

1. **대화 스크립트 포맷**: 노드 기반 JSON — `{ id, speaker, text, choices[], flags[], trustDelta }`. 에디터 없이 Git으로 리뷰 가능.
2. **기억 마스킹 시스템**: 모든 고유명사/로그를 `MemoryToken`으로 래핑 → 칩 상실 시 토큰 단위로 마스킹. UI 전역에 일관 적용.
3. **미니게임 공통 프로토콜**: `protocol CourseGame { func start(stake:), var result: GameResult }` — 7개 게임이 동일 인터페이스로 코스 FSM에 플러그인.
4. **세이브**: 챕터 단위 오토세이브 + 데스게임 직전 체크포인트. 회차 플레이용 클리어 데이터 분리.
5. **로컬라이제이션**: 한국어 우선, String Catalog로 영어/일본어 확장 대비.

## 5. 아트 & 사운드 방향

- **아트 스타일**: 다크 아르누보 + 만찬 테마 (촛불, 은식기, 붉은 벨벳). 2D 일러스트 스탠딩 + 정적 배경 + Live2D급 연출은 후순위.
- **시그니처 비주얼**: 기억이 지워질 때의 "잉크가 번지듯 사라지는" 이펙트.
- **UI 모티브**: 메뉴판/코스 카드 형태의 UI, 기억 칩은 와인 봉인 왁스 모양.
- **사운드**: 코스별 클래식 편곡 BGM, 심리전 구간에서 심박음 레이어 추가.
- **필요 에셋 견적 (MVP)**: 캐릭터 스탠딩 8인 × 표정 4종, 배경 10컷, CG 12컷, UI 세트, BGM 8곡, SFX 40종.

## 6. 개발 로드맵 (약 6개월, 1~2인 기준)

### Phase 0 — 프리프로덕션 (2주)
- [ ] 세계관 바이블 & 7코스 게임 규칙 확정
- [ ] 프로토타입 대상 결정: **코스 1 (기억의 잔) + 대화 시스템**
- [ ] Xcode 프로젝트/SPM 모듈 셋업, CI(GitHub Actions + xcodebuild test)

### Phase 1 — 코어 프로토타입 (4주) ★ 재미 검증 게이트
- [ ] 대화 엔진 + 선택지/신뢰도 시스템
- [ ] 미니게임 1종 (셸 게임) 플레이어블
- [ ] 기억 칩 배팅 → 승패 → 기억 마스킹 루프 1사이클 완성
- [ ] **게이트: "칩을 잃어 정보가 지워지는 경험"이 재미있는가? 아니면 메카닉 피벗**

### Phase 2 — 버티컬 슬라이스 (6주)
- [ ] 프롤로그 + 코스 1~2 스토리 완성 (임시 아트)
- [ ] 미니게임 2종 추가 (거짓의 수프, 침묵의 경매)
- [ ] 단서 수첩, 세이브/로드, 타이틀 화면
- [ ] 외부 테스터 5~10인 플레이테스트

### Phase 3 — 프로덕션 (8주)
- [ ] 코스 3~7 미니게임 + 시나리오 전체
- [ ] 엔딩 4종 분기 구현
- [ ] 정식 아트 에셋 교체, BGM/SFX 통합
- [ ] 기억 소실 연출(MemoryFX) 폴리싱

### Phase 4 — 폴리싱 & 출시 (4주)
- [ ] 밸런싱 (미니게임 난이도 곡선, 칩 경제)
- [ ] TestFlight 베타 (외부 50인)
- [ ] 앱스토어 심의 대응 (등급 설정, 스크린샷/프리뷰 영상)
- [ ] App Store 출시 + ASO (키워드: 데스게임, 추리, 스토리 게임)

## 7. 수익 모델 & 출시 전략

- **기본**: 프리미엄 유료 (₩6,900~9,900) 또는 프롤로그+코스2까지 무료 → 풀버전 IAP 언락 (스토리 게임 검증된 모델).
- **광고 없음** — 몰입형 스릴러와 광고는 상극.
- **추가 IAP(선택)**: 아트북/사운드트랙, 회상 씬 전체 해금권 정도로 최소화.
- **사전 마케팅**: 트레일러(기억 소실 연출 중심), X/유튜브 쇼츠에 "코스 1 데모" 클립.

## 8. 리스크 & 대응

| 리스크 | 대응 |
|---|---|
| 미니게임 7종 = 사실상 게임 7개 개발 | 공통 프로토콜 + 코스 4·5는 기존 메카닉 변형으로 원가 절감. 최악 시 5코스로 축소 |
| 시나리오 분량 과다 | 회상 씬은 텍스트+정적 CG 위주로 제작비 제한 |
| 데스게임 소재의 심의 | 유혈 배제, "퇴장" 연출은 상징적으로 (의자가 비워지는 연출 등) |
| 1인 개발 시 아트 병목 | Phase 2까지 플레이스홀더, 외주는 Phase 3에 일괄 발주 |
| 재미 불확실성 | Phase 1 게이트에서 코어 루프 검증 실패 시 피벗 (예: 멀티플레이 심리전 방향) |
